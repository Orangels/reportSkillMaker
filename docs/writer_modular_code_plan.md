# Writer 模块化代码生成方案

## 问题背景

Kimi（sglang 服务）在执行 wr4/wr5 时，需要一次性生成大量 Python 代码作为 Write 工具的 JSON 参数。代码中包含大量引号、换行、特殊字符，加上历史上下文已经很长，导致 JSON 序列化出错，litellm 报错。

## 根本原因

单次 Write 调用的代码量过大 → JSON 参数过长 → Kimi 生成格式错误概率升高。

## 解决思路

保持 8 步结构不变，在 wr4/wr5 步骤内部要求分模块写入，每次 Write 调用只输出一个小文件（≤150 行）。

## 修改范围

仅修改 `skills/report-gen/guides/writer.md`，共 4 处：

### 修改点1：步骤4（格式设置部分）

**当前**：要求一次性写出格式设置代码

**改为**：分模块写入
- 第一次 Write：格式工具函数文件（set_run_font、set_paragraph_format 等通用函数）
- 第二次 Write：格式配置文件（从模板分析提取的各内容类型格式参数映射表）
- 每个文件独立 Write 调用

### 修改点2：步骤5（内容仿写部分）

**当前**：要求编写内容仿写代码（一次性输出）

**改为**：分模块写入
- 根据步骤3规划的报告结构，将内容生成代码拆分为多个模块文件
- 每个模块文件不超过 150 行
- 最后编写入口脚本，import 各模块，组装完整报告并输出

### 修改点3：TodoWrite wr4 描述

**当前**：
```
"编写脚本格式部分 → set_run_font 必须包含 font.size/color.rgb/bold/name（禁止省略任何属性），行距用 twips 原值"
```

**改为**：
```
"编写脚本格式部分 → 分模块写入：格式工具函数文件 + 格式配置文件，每个文件独立 Write，set_run_font 必须包含 font.size/color.rgb/bold/name，行距用 twips 原值"
```

### 修改点4：TodoWrite wr5 描述

**当前**：
```
"编写脚本内容部分 → 按模板结构逐章节仿写 + 段内关键词加粗用多 run 实现"
```

**改为**：
```
"编写脚本内容部分 → 按报告结构分模块写入（每个模块文件 ≤150 行），最后编写入口脚本组装，段内关键词加粗用多 run 实现"
```

## 泛化性检查

| 检查项 | 结果 |
|--------|------|
| 是否写死了模块数量 | ❌ 没有，由报告结构决定 |
| 是否写死了文件名 | ❌ 没有，由 Writer 自行命名 |
| 是否写死了章节拆分方式 | ❌ 没有，引用步骤3的结构规划 |
| 是否绑定特定模板 | ❌ 没有 |
| 核心约束是否通用 | ✅ "单文件 ≤150 行" + "分多次 Write" 适用任何模板 |

## 不改的部分

- 步骤数仍为 8 步
- wr6（执行脚本）— 改为执行入口脚本，现有描述已覆盖
- wr7/wr8 — 不受影响
- 其他文件（SKILL.md、data_expert.md、template_analyst.md）— 不受影响
