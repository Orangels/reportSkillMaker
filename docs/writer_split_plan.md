# Writer 拆分方案

## 问题

单 Writer agent 顺序执行 wr1-wr8，到后半段 input_tokens 达到 85K+。report_plan.md 在 wr3（~28K 处）生成后，被后续代码输出不断推远：

- wr5 编码后半段：plan 在 ~40K tokens 前 → 补充章节整章缺失
- wr7 验证阶段：plan 在 ~60K tokens 前 → data_usage_check.md 未生成、自检失效

实测结果：第三层+主动发现数据利用率 0%，报告内容只覆盖了 plan 的前半部分。

## 根因

plan 在上下文中段（28%位置），随步骤推进沉入注意力最弱区域。核心问题不是上下文总长度，而是**关键信息在上下文中的位置**。

## 方案：拆为 3 个 Agent

```
Team Lead (SKILL.md 步骤6)
  ├─ Writer-Planner    wr1+wr2+wr3  → report_plan.md（增强版）
  ├─ Writer-Coder      wr4+wr5+wr6  → .py files + .docx
  └─ Writer-Verifier   wr7+wr8      → data_usage_check.md + 验证结论
```

### Agent 1: Writer-Planner（规划）

**读入：**
- analysis_template.md
- extracted_data.json
- template_content.md

**执行：**
- wr1：从 TA 提取格式规范数值（字号、颜色、加粗、字体）
- wr2：全量盘点 DE 数据维度，理解 TA 框架逻辑，检查数据完整性，学习模板写法
- wr3：规划报告结构 + 数据消化方案

**输出：**
- `report_plan.md`（增强版）

**上下文估算：** ~40K tokens

**拆分理由：** 规划阶段需要同时持有 TA + DE + template_content 三份大文件，但产出是一份紧凑的 plan。后续 agent 只需要读 plan，不需要再读原始文件。

### Agent 2: Writer-Coder（编码+执行）

**读入：**
- report_plan.md（增强版，含格式速查表、章节清单、消化去向表）
- extracted_data.json

**不再读入：** analysis_template.md、template_content.md（有用信息已在 Planner 阶段消化进 report_plan.md）

**执行：**
- wr4：写格式工具函数 + 格式配置（从 plan 的格式速查表取值）
- wr5：按 report_plan.md 的编码章节清单逐章编码，每完成一个章节模块对照清单打钩
- wr6：写入口脚本，执行生成 docx

**输出：**
- 会话目录下的 .py 文件
- 最终 .docx 文件

**上下文估算：** 起步 ~20K（plan ~5K + DE ~15K），编码过程增长到 ~62K

**拆分理由：** plan 从上下文中段（28%位置）移到开头（0-8%位置），处于 transformer U 型注意力曲线的最强区域。plan 前无噪声，是 Agent 2 接触到的第一份信息。

### Agent 3: Writer-Verifier（验证）

**读入：**
- report_plan.md
- extracted_data.json
- 生成的 docx（通过 pandoc 转 md 后读入）

**执行：**
- wr7：
  1. 格式抽查（字号/颜色/加粗/Markdown 泄漏）
  2. 逐维度对照 plan 的消化去向表，输出 `data_usage_check.md`
  3. 重点对象深度验证（≥3 维度）
  4. 检查是否有整章缺失
- wr8：确认文件保存

**输出：**
- `data_usage_check.md`
- 验证结论（通过 / 不通过 + 具体缺陷列表）

**上下文估算：** ~30K tokens

**拆分理由：** 验证需要同时对照 plan 和 DE 逐维度检查。独立 agent 上下文新鲜（~30K），注意力满格。

## 关键信息位置对比

| 指标 | 拆分前（单 Writer） | 拆分后（Writer-Coder） |
|------|---------------------|----------------------|
| plan 在上下文中的位置 | 中段（28%处） | **开头（0-8%处）** |
| plan 前面的噪声 | 23K（TA+DE+template_content 阅读过程） | 0 |
| wr5 末尾时 plan 距离 | ~40K tokens 前 | ~32K tokens 前 |
| wr7 验证时 plan 距离 | ~60K tokens 前 | ~5K tokens 前（独立 agent 重新读入） |
| 总上下文峰值 | ~85K | Coder ~62K / Verifier ~30K |

## 增强版 report_plan.md

拆分后 report_plan.md 成为 3 个 Agent 间的**唯一通信协议**。Writer-Coder 和 Writer-Verifier 不再读取 analysis_template.md 和 template_content.md，所有必要信息必须在 plan 中自包含。

### 风险评估

| 优先级 | 风险 | 严重性 | 理由 |
|--------|------|--------|------|
| P0 | 编码章节清单不完整 | 致命 | 直接导致整章缺失，本次测试的核心问题 |
| P0 | 消化去向表精度不够 | 致命 | Coder 找不到数据或遗漏维度 |
| P1 | 格式规范速查表缺失精确值 | 严重 | 格式全错，但内容至少在 |
| P1 | 段落写法规则传递 | 严重 | 风格不符，但数据至少用了 |
| P2 | plan 体量膨胀 | 中等 | 可控，有明确上限即可 |

### plan 内容结构（7 个模块）

#### 模块1：格式规范速查表（必填，~500 tokens）

从 TA 提取具体数值，Writer-Coder 直接使用，不用再翻 TA 原文。

**约束：每个字段必须是具体数值，禁止"参见 TA"引用。**

```
| 内容类型 | 字体 | 字号 | 颜色 | 加粗 | 行距 |
|---------|------|------|------|------|------|
| 红头 | 方正小标宋简体 | 55pt | FF0000 | 否 | 不设限制 |
| 主标题 | 方正小标宋简体 | 22pt | 默认黑色 | 否 | 560twips |
| 一级标题 | 黑体 | 16pt | 默认黑色 | 否 | 560twips |
| ... | ... | ... | ... | ... | ... |
```

#### 模块2：段内加粗规则（必填，~200 tokens）

从 TA 的加粗模式分析中提炼。

```
加粗对象：[从 TA 提取的加粗规律，如类别名称、序号标题、引导词等]
实现方式：多 run，bold=True 控制，禁止 Markdown ** 标记
```

#### 模块3：编码章节清单（必填，~300 tokens）

Writer-Coder 逐章编码并打钩，防止遗漏。

**粒度规则：**
- 一级标题必列
- 二级标题必列
- 每个分析对象独立一行
- 重点标注和维度数要求直接写在清单中
- 落款、抄送独立一行

```
[ ] 首版红头 + 主标题
[ ] 一、整体情况
[ ] 二、上升警情类别分布 - 章节标题
[ ] （一）道路交通类警情分析（重点，≥3维度）
[ ] （二）社会联动警情分析（一般）
[ ] （三）投诉监督警情分析（一般）
[ ] 三、下降警情类别分布 - 章节标题
[ ] （一）群众紧急求助警情分析（重点，≥3维度）
[ ] 补充：涉未成年人警情分析（重点敏感，≥4维度）
[ ] 补充：涉刀警情分析（重点敏感，≥4维度）
[ ] 补充：治安类警情详细分析（补充，≥3维度）
[ ] 四、工作建议 - 共5条
[ ] 落款 + 抄送
```

#### 模块4：章节大纲 + 分析对象维度列表（必填，~1500 tokens）

每个分析对象列出计划使用的维度，并标注 DE JSON 路径，让 Coder 能直接定位数据。

```
（一）道路交通类警情分析（重点）：
  - 警情类型分析 → DE: 第二层_多维度分析.道路交通类警情.分析维度.警情类型分析
  - 发案时间分析 → DE: 第二层_多维度分析.道路交通类警情.分析维度.发案时间分析
  - 时段类型分析 → DE: 第二层_多维度分析.道路交通类警情.分析维度.时段类型分析
  - 辖区分布分析 → DE: 第二层_多维度分析.道路交通类警情.分析维度.辖区分布分析
  - 处理结果分析 → DE: 第二层_多维度分析.道路交通类警情.分析维度.处理结果分析
  - 星期分布补充 → DE: 主动发现数据.星期分布
```

**精度说明：** 精确到"章节+维度角色+DE JSON路径"。不需要精确到段落序号，Coder 在模块内部自行决定段落顺序。

#### 模块5：消化去向汇总表（必填，~500 tokens）

覆盖 DE 的**所有**数据节点（第一层、第二层、第三层、主动发现），逐项标注去向或不用理由。

```
| DE数据节点 | 计划去向 | 使用方式 |
|-----------|---------|---------|
| 第一层.有效警情总数 | 第一章 | 总述数据 |
| 主动发现.星期分布 | 第二章(一) | 补充周末效应 |
| 主动发现.重复报警 | 不使用 | 数据量小（23个号码），无分析价值 |
```

#### 模块6：段落写法规则（必填，~400 tokens）

从 template_content.md 提炼为可执行的句式模板和结构模式，不照搬原文。

```
段落写法规则：

1. 总述句型："我局共接报**{类别名}**{数量}起，环比{升降}{比率}%。"
2. 分析维度句型："**{序号}.从{维度名}分析。**主要集中在{项目1}（{数量}起，占比{比率}%），其次{项目2}{数量}起、{项目3}{数量}起。"
3. 小结模式："**小结：**本月{类别}呈{趋势}态势，{特征概述}：**一是**{特征1}；**二是**{特征2}；**三是**{特征3}。"
4. 建议模式：标题"动词+名词"，内容含责任单位+具体措施+量化要求+目标导向
5. 数字规范：数量用阿拉伯数字，百分比保留1位小数
```

#### 模块7：分析对象重要程度标注（必填，~200 tokens）

```
| 分析对象 | 重要程度 | 引用DE维度数规划 |
|---------|---------|-----------------|
| 道路交通类 | 重点 | ≥3 |
| 涉未成年人 | 重点敏感 | ≥4 |
| ... | ... | ... |
```

### 体量控制

| 模块 | 预估 tokens |
|------|------------|
| 格式规范速查表 | ~500 |
| 段内加粗规则 | ~200 |
| 编码章节清单 | ~300 |
| 章节大纲+维度列表（含 DE 路径） | ~1500 |
| 消化去向汇总表 | ~500 |
| 段落写法规则 | ~400 |
| 重要程度标注 | ~200 |
| **合计** | **~3600** |

**上限 4K tokens。** 增强版主要是替换而非追加——格式速查表替代"参见 TA"引用，编码清单替代纯大纲，写法规则替代模板原文引用。去掉对 Coder 无用的内容（如"生成文件清单"）。

### Planner 输出质量保障

1. **writer_planner.md 中定义 plan 输出模板** — 给 Planner 一个骨架模板，每个模块标注"必填"和格式要求，按模板填充而非自由发挥
2. **Team Lead 验证 plan 质量** — 在调用 Writer-Coder 之前检查：
   - 格式速查表是否有具体数值（无"参见"引用）
   - 编码清单是否覆盖所有章节
   - 消化去向表是否覆盖 DE 所有数据节点
   - 重点对象是否标注且维度数 ≥3
3. **Planner 的 TodoWrite 强制提醒** — wr3 的 TodoWrite 描述中列出 plan 必须包含的 7 个模块

## 对本次测试报告问题的预期改善

| 问题 | 原因 | 拆分后改善 |
|------|------|----------|
| 补充章节整章缺失（涉未成年、涉刀） | plan 在 70K 前已遗忘 | plan 在 Coder 上下文开头，编码清单逐章打钩 |
| 数据利用率 ~30% | wr5 后半段丢失消化去向记忆 | 消化去向表在 Coder 上下文开头 |
| data_usage_check.md 未生成 | wr7 在 85K 时注意力为零 | Verifier 独立 agent，上下文 30K 新鲜 |
| Markdown **泄漏到 docx | wr7 验证被跳过 | Verifier 能可靠检测 |
| 建议只有3条缺2条 | plan 末尾建议被遗忘 | 编码清单明确列出5条 |
| 小结模式不符合模板 | template_content 写法学习在 70K 前 | Planner 将写法规则消化进 plan |

## 风险点及应对

| 风险 | 评估 | 应对 |
|------|------|------|
| Agent 间信息损耗 | **已通过增强版 plan 解决** — 7 个模块覆盖 Coder/Verifier 所需的全部信息，无需读原始文件 | Planner 输出模板 + Team Lead 验证 plan 质量 |
| report_plan.md 膨胀 | **可控** — 增强版 ~3.6K tokens，主要是替换非追加，上限 4K | 去掉对 Coder 无用内容，体量可能比现在更紧凑 |
| 3次 agent 调用时间成本 | **可接受** — 总 token 消耗相近，换来内容完整性大幅提升 | Planner 和 Verifier 上下文小，执行快 |
| Verifier 修补能力有限 | **设计如此** — Verifier 只发现问题，修复由 Team Lead 决定是否重调 Coder | Team Lead 根据缺陷严重程度决定是否重试 |
