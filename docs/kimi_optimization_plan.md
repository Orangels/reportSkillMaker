# Kimi 模型优化方案：借鉴 Opus 的 TodoWrite 模式

## 一、问题背景

### Opus 的 TodoWrite 使用模式（两层结构）

**第一层：主流程顶层 TodoWrite**
Opus 在执行 report-gen skill 时，会自动创建一个顶层进度清单：
```
TodoWrite([
  { id: "1", content: "参数收集与初始化", status: "in_progress" },
  { id: "2", content: "调用 Template Analyst 分析模板", status: "pending" },
  { id: "3", content: "调用 Data Expert 提取数据", status: "pending" },
  { id: "4", content: "调用 Writer 生成报告", status: "pending" }
])
```
每完成一个步骤就更新状态，起到进度追踪和防跳步作用。

**第二层：Subagent 内部 TodoWrite**
每个 subagent 在开始执行时，也会自动创建自己的步骤清单。例如 Template Analyst 会创建：
```
TodoWrite([
  { id: "1", content: "读取模板内容", status: "pending" },
  { id: "2", content: "识别文档类型", status: "pending" },
  { id: "3", content: "结构分析", status: "pending" },
  { id: "4", content: "内容逻辑分析", status: "pending" },
  { id: "5", content: "格式规范分析", status: "pending" },
  { id: "6", content: "语言风格分析", status: "pending" },
  { id: "7", content: "生成提取清单", status: "pending" },
  { id: "8", content: "保存分析结果", status: "pending" },
  { id: "9", content: "验证检查清单", status: "pending" }
])
```

### Kimi 的现状

- Kimi 在 Claude Code 环境中运行，**可以调用 TodoWrite 工具**
- 但 Kimi 不会像 Opus 那样主动使用 TodoWrite
- 导致执行过程缺少纪律约束，容易跳步、遗漏验证

### TodoWrite 的核心价值

1. **执行纪律**：强迫模型在动手前先列出完整步骤，防止跳步
2. **进度追踪**：每完成一步更新状态，形成自我检查机制
3. **防遗漏**：未完成的 todo 项是明确的提醒，不会被遗忘

### Kimi Data Expert 的数据质量差距（中间文件对比分析）

通过对比 Opus 和 Kimi 生成的 `extracted_data.json`，发现关键质量差距：

| 对比维度 | Opus（86KB, 3273行） | Kimi（22KB, 1057行） |
|---------|---------------------|---------------------|
| 数据组织方式 | 按报告章节层级化组织 | 扁平列表罗列 |
| 多维度交叉分析 | 每个警情类型独立统计辖区分布、时段分布 | 仅全局统计，无按类型拆分 |
| 子类展开深度 | 重点警情（殴打、盗窃）有子类拆分 | 只有顶层分类计数 |
| 特殊分析项 | 涉刀警情（含具体案例）、敏感警情 | 完全缺失 |
| 数据探查轮次 | 4-5轮探查后编写提取脚本 | Data Expert 失败，Writer 临时提取 |

**根本原因**：Kimi 的 Data Expert 没有严格按照模板分析的"数据提取清单"逐项提取。它做的是"把 Excel 里能看到的字段都列出来"的数据罗列，而不是"按照报告需要的维度组织数据"的目标导向提取。

**具体表现**：
1. **缺少多维度交叉分析**：没有按警情类型分别统计辖区分布、时段分布
2. **缺少深层子类展开**：没有对重点警情（殴打、盗窃）做子类拆分
3. **缺少特殊分析项**：涉刀警情、敏感警情等完全缺失
4. **数据组织不匹配报告结构**：扁平列表无法直接支持报告的层级化仿写

### Kimi 执行流程问题（Log 对比分析）

通过对比两个模型的完整执行 log，发现 4 个关键流程差异：

**关键差异1：Data Expert 阶段 — 致命失败**

| 对比项 | Opus Data Expert | Kimi Data Expert |
|--------|-----------------|-----------------|
| 探查脚本数量 | 5个（逐步深入） | 1个（第一次就报错） |
| 提取脚本大小 | 22KB | 未产出（失败） |
| 工具调用次数 | 大量（多轮探查+提取） | 仅7次，149秒就返回 |
| 产出文件 | 86KB extracted_data.json | 无（未生成） |
| 失败原因 | — | Python 环境报错（venv 无 pip，pandas 装在 anaconda） |
| 重试策略 | — | 尝试2次修复后放弃 |

**关键差异2：主 Agent 验证缺失**

Opus 主 Agent：Data Expert 完成后检查了 extracted_data.json 是否存在且大小合理。

Kimi 主 Agent：Data Expert 返回后，**没有验证输出文件**，直接说"数据提取完成"就启动了 Writer。

```
# Kimi log 关键行：
06:40:48 | Agent:25 返回（total_tokens: 34894, tool_uses: 7）  ← 只用了7次工具就返回
06:40:55 | "数据提取完成！让我验证数据文件："  ← 说要验证但没做
06:41:15 | 直接调用 Writer subagent  ← 跳过验证，直接启动下一步
```

**关键差异3：Writer 被迫做了 Data Expert 的活（角色错位）**

Kimi 的 Writer 启动后：
1. 尝试读取 extracted_data.json → 文件不存在
2. 被迫自己写 extract_august_data.py 做数据提取
3. 遇到同样的 Python 环境问题，多次重试后用 anaconda python 跑通
4. 但遇到 KeyError（列名猜错），修正后产出 extracted_data.json
5. 因为 Writer 不是数据专家，提取结果是扁平化的分类计数，缺少多维度交叉分析

**关键差异4：数据探查深度**

| 指标 | Opus Data Expert | Kimi Writer（代做） |
|------|-----------------|-------------------|
| 探查脚本数量 | 5个（逐步深入） | 1个（一步到位） |
| 提取脚本大小 | 22KB | 7KB |
| 了解列名 | 通过多轮探查充分了解 | 直接猜测列名，出错后修正 |
| 辖区分布 | 每类警情独立统计 | 只有全局统计 |
| 时段分布 | 有 | 无 |
| 涉刀/敏感分析 | 有（含案例详情） | 无 |

### 根因总结

Kimi 报告质量差的完整因果链：

```
Data Expert Python环境失败（尝试2次就放弃）
  → 返回时没有产出 extracted_data.json
    → 主Agent没有验证就启动Writer（验证缺失）
      → Writer发现没有数据文件
        → Writer被迫自己做数据提取（角色错位）
          → 提取逻辑简单（扁平列表，缺少多维度）
            → 最终报告缺乏深度
```

**核心问题归纳（7个）：**
1. **P1 - 缺少执行纪律**：Kimi 不主动使用 TodoWrite，容易跳步
2. **P2 - 主 Agent 验证缺失**：没有检查中间产出文件就进入下一步
3. **P3 - Python 环境适应能力弱**：Kimi 的 subagent 尝试次数少就放弃了
4. **P4 - 数据提取不按清单**：Data Expert 没有严格按清单逐项提取，结果扁平化
5. **P5 - 角色错位无补救**：Writer 做了 Data Expert 的活，但能力不足
6. **P6 - 数据探查深度不够**：1轮探查 vs Opus 5轮渐进式探查
7. **P7 - 注意力衰减**：Kimi 在长对话中对指导文档的遵循度下降，越往后越容易跳步

## 二、修改方案

### 2.0 修改策略：只追加，不修改现有内容

**核心原则**：现有指导文档的内容已经过验证，Opus 使用效果良好。为避免改动引入新问题，采用**纯追加策略**：

1. **不修改**现有指导文档的任何内容（不删除、不替换、不重组）
2. **只在文档末尾追加**新的章节
3. **通过 TodoWrite 动态加载**机制，让模型在执行每个步骤时重新阅读指导文档的对应章节

**为什么用"动态加载"而非"合并替换"**：
- 合并替换可能删除有用内容（如"利用结构化字段"这条关键原则）
- Kimi 在长对话中注意力衰减（P7），需要在每个步骤强制重新加载相关章节
- TodoWrite 不仅是进度追踪工具，更是**注意力导向工具**和**知识索引工具**

**TodoWrite 的三重角色**：
1. **进度追踪**：防跳步、防遗漏
2. **注意力导向**：每个步骤的"【加载】"指令强制模型重新聚焦到相关章节
3. **知识索引**：TodoWrite 清单本身就是指导文档的结构化索引

#### 追加内容2：数据提取质量要求

```markdown
## 数据提取质量要求（必须遵守）

### 1. 强制对照清单逐项提取
- 阅读 analysis_template.md 中的"数据/信息提取清单"
- 将清单中的每一项作为一个独立的提取任务
- 禁止"凭感觉提取"，必须逐项对照

### 2. 层级化输出结构
输出的 extracted_data.json 必须按报告章节层级组织，而非扁平列表。
```

### TodoWrite 动态加载机制说明

**问题**：Kimi 在长对话中（执行多个步骤后），对指导文档前面章节的遵循度下降。

**解决方案**：每个 TodoWrite 步骤包含【加载：章节名】指令，要求模型在执行该步骤前重新阅读对应章节。

**示例**：
```

```
当 Kimi 执行到 de4 时，会被指令驱动去重新阅读：
1. 指导文档中的"步骤4：按清单逐项提取"章节
2. "关键原则"中的"按清单提取"条目
3. 新追加的"数据提取质量要求"章节

这确保了即使在对话后期，模型仍然能获取到完整的执行指导。

### 5.6 真正的按需加载机制（后续优化方向）

当前【加载】机制是"注意力重定向"——整个指导文档在 subagent 首次读取时一次性加载到上下文，【加载】指令只是提示模型重新关注已有内容。这对抗注意力衰减有效，但存在局限：

1. **上下文开销**：即使某些章节在当前步骤不需要，也占用了 token
2. **依赖模型自觉**：模型是否真的"重新阅读"了指定章节，无法保证

**真正的按需加载方案**：将指导文档拆分为独立文件，TodoWrite 步骤中改为显式 Read 指令。

```
当前（注意力重定向）：
  指导文档一次性加载 → TodoWrite 步骤提示"重读步骤4章节"

优化后（按需加载）：
  主文档只保留概览和 TodoWrite 清单
  → 执行到 de4 时，显式 Read [SKILL_DIR]/guides/data_expert/step4_extract.md
  → 模型获得该步骤的完整指导，且是"新鲜"的上下文
```

**拆分示例（以 data_expert 为例）**：
```
guides/
├── data_expert.md                    ← 精简为概览 + TodoWrite 清单
└── data_expert/                      ← 按步骤拆分的详细指导
    ├── step1_read_analysis.md        ← 步骤1详细指导
    ├── step2_explore_data.md         ← 步骤2详细指导（含渐进式探查要求）
    ├── step3_check_capability.md     ← 步骤3详细指导
    ├── step4_extract.md              ← 步骤4详细指导（含数据提取质量要求）
    ├── step5_validate.md             ← 步骤5详细指导（含验证清单）
    └── step6_save.md                 ← 步骤6详细指导
```

**TodoWrite 清单改写示例**：
```
{ id: "de4", content: "按清单逐项提取 → Read [SKILL_DIR]/guides/data_expert/step4_extract.md 后执行", status: "pending" }
```

**预期收益**：
- 每步只加载必要的指导内容，节省上下文 token
- Read 操作是确定性的（不依赖模型自觉），保证指导内容被实际加载
- 可以为每步提供更详细的指导，不受主文档长度限制

**实施条件**：当前方案经过测试验证后，如果仍存在"模型未遵循【加载】指令"的问题，再实施此优化。
