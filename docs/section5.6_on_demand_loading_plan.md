# Section 5.6 按需加载优化计划

## 一、背景

### 1.1 前置依赖

本文档是 `kimi_optimization_plan.md` Section 5.6 的详细实施方案。前置方案（Section 二）已全部完成并打 tag `kimi_skill_v1.0`。

### 1.2 当前问题

| 问题 | 现状 | 影响 |
|------|------|------|
| 指导内容位置劣势 | subagent 启动时一次性读取整个指导文档（191-225行），后续步骤执行时该内容已被推到上下文深处 | 后期步骤的指导内容远离模型注意力焦点 |
| 【加载】靠模型自觉 | "重新阅读步骤X章节"是提示，不是强制动作 | 无法保证模型真的回看了对应内容 |

**重要说明**：本方案的核心价值是**注意力位置优化 + 确定性加载**，而非上下文总量减少。由于步骤文件在同一 subagent 对话中逐步累积，执行到最后一步时上下文总量与未拆分版本基本一致。真正减少上下文总量需要更激进的架构变化，参见"十、后续优化方向"。

### 1.3 核心思路

将每个指导文档拆分为：**精简主文件（概览 + TodoWrite 清单）** + **按步骤独立的子文件**。

TodoWrite 中的【加载】指令从"提示模型回看"改为"显式 Read 工具调用"，确保每步指导以新鲜上下文出现在对话末尾。

## 二、核心设计

### 2.1 改造前后对比

**改造前**：
```
subagent 启动 → Read data_expert.md (191行全量加载) → 执行 de1-de8 (靠【加载】提示回看)
```

**改造后**：
```
subagent 启动 → Read data_expert.md (精简版 ~55行，含概览+TodoWrite清单)
              → 执行 de1: Read steps/de1_read_analysis.md (~15行)
              → 执行 de2: Read steps/de2_explore_data.md (~25行)
              → ...每步按需加载
```

### 2.2 关键收益

- **确定性加载**：Read 工具调用是确定性的，不依赖模型自觉"回看"
- **注意力位置优化**：每步指导以新鲜上下文出现在对话末尾（注意力最强的位置），而非埋在对话开头被数千行执行过程淹没
- **初始加载量降低 ~75%**：首次 Read 从 ~200行降至 ~50行，模型在开始执行前对全局有清晰认知而不被细节干扰
- **步骤文件自包含**：每个文件包含该步骤所需的全部信息，无需跨章节查找

**注意：上下文总量不会减少。** 随着步骤逐步执行，Read 的步骤文件在上下文中累积。执行到 de7 时，de1-de6 的步骤文件内容仍在上下文中，累积总量与未拆分版本基本一致。本方案的价值在于"当前步骤的指导内容在对话末尾"，而非"总量更少"。

### 2.3 TodoWrite 模板改写示例

**改造前**：
```
{ id: "de4", content: "【加载：步骤4+数据提取质量要求】第一层提取：总量统计、分类汇总、环比同比、占比计算", status: "pending" }
```

**改造后**：
```
{ id: "de4", content: "第一层提取 → 先执行 Read [SKILL_DIR]/guides/data_expert/steps/de4_extract_layer1.md，然后按文件指导提取", status: "pending" }
```

## 六、上下文量分析

### 6.2 上下文累积过程（以 data_expert 为例）

```
执行阶段        当前步骤新加载    累积指导内容总量    vs 未拆分版本(191行)
──────────────────────────────────────────────────────────────
启动            主文件 ~55行      ~55行              29% ✅ 显著减少
de1             +15行            ~70行              37% ✅
de2             +25行            ~95行              50% ✅
de3             +10行            ~105行             55% ✅
de4             +20行            ~125行             65%
de5             +20行            ~145行             76%
de6             +20行            ~165行             86%
de7             +30行            ~195行             102% ❌ 基本持平
de8             +15行            ~210行             110% ❌ 略超（步骤文件有少量重复）
```

**结论**：前半程（de1-de4）上下文量显著少于未拆分版本；后半程（de7-de8）累积量基本持平甚至略超。但**核心收益不在总量**，而在于当前步骤的指导内容始终出现在对话末尾的注意力焦点位置。

### 6.3 与消费者文件拆分（Section 11）的组合效果

Section 11 将分析文件从 ~350行完整版缩减为消费者专属版（DE ~80行，Writer ~260行），这是**真实的上下文总量减少**。以 Data Expert 为例，两项优化叠加后：

```
优化项                       改造前        改造后        减少量
─────────────────────────────────────────────────────────
分析文件输入（Section 11）   ~350行        ~80行         -270行（真实减少）
指导文档初始加载（Section 5.6）~191行      ~55行         注意力位置优化
─────────────────────────────────────────────────────────
DE 启动时总上下文             ~541行        ~135行        -75%
```

两项优化互补：Section 11 减少真实上下文总量，Section 5.6 优化剩余内容的注意力位置。

## 八、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| 步骤文件自包含导致内容重复 | 文件总行数增加 | 可接受：每个步骤文件 < 55行，适度重复换取自包含性 |
| 模型不执行 Read 调用 | 步骤指导未加载 | TodoWrite 中用明确指令"先执行 Read xxx"；主文件执行纪律章节强调 |
| 步骤文件路径写错 | Read 失败 | 使用 `[SKILL_DIR]` 变量，与现有路径机制一致 |
| 共享内容修改时需同步多个文件 | 维护成本增加 | 共享内容量有限（数据提取质量要求仅 7行），且各步骤文件只包含该步骤相关的子集 |
| SKILL.md subagent prompt 需更新 | 联动修改 | 阶段一步骤2 已覆盖：更新 DE/Writer 的分析文件路径 |

## 十、后续优化方向：真正减少上下文总量

本文档包含两类优化：指导文档按需加载（Section 二至六）解决**注意力位置**问题，中间产物消费者拆分（Section 十一）解决**分析文件的上下文总量**问题。如果测试后发现**指导文档在同一 subagent 对话中累积**导致后期步骤质量下降，需要考虑以下更激进的架构变化。

### 方案 A：每步独立 subagent

**做法**：将 de1-de8 每步都作为独立的 subagent 调用，由主流程（或一个中间协调 agent）串联。

```
主流程/协调Agent
├─ subagent: de1（上下文：主文件55行 + de1步骤文件15行 = ~70行）
│  └─ 输出：de1_result.json（或更新共享状态文件）
├─ subagent: de2（上下文：主文件55行 + de2步骤文件25行 + de1结果摘要 = ~100行）
│  └─ 输出：de2_result.json
├─ ...
└─ subagent: de8（上下文：主文件55行 + de8步骤文件15行 + 前序结果摘要 = ~120行）
   └─ 输出：extracted_data.json
```

**效果**：
- 每步上下文量恒定（主文件 + 当前步骤文件 + 前序结果摘要），不随步骤推进增长
- 彻底消除累积问题

**代价**：
- 8 次 subagent 调用开销大（每次启动有固定成本）
- **步骤间状态传递复杂**：de2 需要 de1 的发现结果，de4 需要 de2/de3 的探查结果，需要设计中间状态文件格式
- 每个 subagent 对数据源文件（Excel）的理解不连续，可能导致重复探查
- 调试困难：出问题时需要追踪多个 subagent 的输出

**适用场景**：模型上下文窗口极小、或步骤间依赖很弱的情况。

### 方案 B：分阶段 subagent

**做法**：将 8 步分为 2-3 个阶段，每阶段一个 subagent，阶段内步骤仍在同一对话中顺序执行。

```
主流程
├─ subagent: 阶段1-探查（de1 + de2 + de3）
│  └─ 输出：data_exploration_report.md（数据源结构、字段映射、能力评估）
├─ subagent: 阶段2-提取（de4 + de5 + de6）
│  └─ 输入：data_exploration_report.md
│  └─ 输出：extracted_data.json
└─ subagent: 阶段3-验证保存（de7 + de8）
   └─ 输入：extracted_data.json
   └─ 输出：validated_extracted_data.json
```

**效果**：
- 每阶段上下文约为总量的 1/3
- 阶段间通过结构化文件传递状态，断点清晰
- 兼顾了上下文控制和步骤连续性

**代价**：
- 需要定义阶段间的中间产物格式（如 data_exploration_report.md）
- 3 次 subagent 调用（比方案 A 少，但比现有方案多 2 次）
- 阶段划分需要仔细设计，确保阶段内步骤的依赖关系紧密、阶段间接口清晰

**推荐的阶段划分**：

| 角色 | 阶段1 | 阶段2 | 阶段3 |
|------|------|------|------|
| data_expert | de1+de2+de3（理解阶段） | de4+de5+de6（提取阶段） | de7+de8（验证保存） |
| writer | wr1+wr2+wr3（规划阶段） | wr4+wr5（编码阶段） | wr6+wr7+wr8（执行验证） |
| template_analyst | ta1+ta2+ta3+ta4（结构分析） | ta5+ta6（格式风格） | ta7+ta8（清单输出） |

**适用场景**：本方案（Section 5.6）测试后发现后期步骤质量仍不理想，需要进一步减少上下文总量。

### 方案对比

| 维度 | 当前方案(v1.0) | Section 5.6(本文档) | 方案A(每步独立) | 方案B(分阶段) |
|------|--------------|-------------------|---------------|-------------|
| 改造复杂度 | — | 低（拆文件） | 高（重构架构） | 中（拆阶段+定义接口） |
| 上下文总量 | 191-225行 | 相同（累积） | ~70-120行/步 | ~80-120行/阶段 |
| 注意力位置 | 差（全埋开头） | 好（每步末尾） | 最好（每步独立） | 好（每阶段独立） |
| 确定性加载 | 无（靠自觉） | 有（显式Read） | 有（独立上下文） | 有（独立上下文） |
| subagent 调用数 | 1次/角色 | 1次/角色 | 8次/角色 | 3次/角色 |
| 状态传递复杂度 | 无 | 无 | 高 | 中 |
| 实施优先级 | ✅ 已完成 | ⬅️ 当前 | 待评估 | 待评估 |

### 推荐路径

```
v1.0（已完成）→ Section 5.6（当前）→ 测试验证 → 如仍有问题 → 方案B（分阶段）
                                                           → 如仍不够 → 方案A（每步独立）
```

方案 B 是 Section 5.6 和方案 A 之间的平衡点，建议作为下一步优先考虑的方向。方案 A 作为最后手段保留。

## 十一、中间产物冗余优化：按消费者拆分分析文件

### 11.1 问题背景

Template Analyst 输出的 `analysis_template.md`（~350行）是 Data Expert 和 Writer 的共同输入，但两者实际需要的章节差异巨大：

| 章节 | 行数 | DE 需要 | Writer 需要 |
|------|------|---------|-------------|
| 一、文档基本信息 | ~30 | ❌ | ✅（受众定位影响语言风格） |
| 二、报告结构框架 | ~35 | ❌ | ✅ |
| 三、格式规范详解 | ~40 | ❌ | ✅ |
| 四、内容逻辑分析 | ~40 | ⚠️ 仅需数据指标体系+验证规则 | ✅ |
| 五、可变元素分析 | ~25 | ❌ | ✅ |
| 六、表达方式和语言风格 | ~50 | ❌ | ✅ |
| 七、数据/信息提取清单 | ~45 | ✅ | ❌ |
| 八、验证检查清单 | ~25 | ❌ | ❌ |
| 九、分析说明 | ~25 | ❌ | ❌ |

**冗余度量化**：
- Data Expert 实际需要 ~60行（约 17%），其余 ~290行 为无关内容
- Writer 实际需要 ~220行（约 63%），其余 ~130行 为无关内容

### 11.2 章节分类决策

#### 第八章（验证检查清单）：两个消费者均不需要

| 子节 | 对 DE | 对 Writer | 结论 |
|------|-------|-----------|------|
| 8.1 数据验证 | 与第四章"数据验证规则"重复，且 DE 自身指导文档已有验证清单 | — | 不纳入 |
| 8.2 格式验证 | — | 直接复述第三章格式规范表中的值 | 不纳入 |
| 8.3 内容验证 | — | `writer.md` 的验证清单更详细且已覆盖 | 不纳入 |

#### 第九章（分析说明）：纯元信息（XML提取来源、中间文件清单），仅用于调试回溯，消费者不需要

#### 第一章（文档基本信息）：仅 Writer 需要

- 受众信息（县级公安局领导层、抄报县政府领导）直接影响语言正式程度和权威性表达
- 文档类型（混合型：统计+分析+建议）帮助 Writer 理解报告的结构平衡
- Data Expert 的工作完全由数据提取清单驱动，不受受众和文档类型影响

### 11.3 方案：Template Analyst 按消费者分开输出

**核心思路**：让 Template Analyst 在分析完成后，直接输出 2 个消费者专属文件（不生成完整版）。

#### 输出文件

```
[会话目录]/
├── analysis_for_data_expert.md       ← DE 专属（~80行）
└── analysis_for_writer.md            ← Writer 专属（~260行）
```

#### 内容分配

**`analysis_for_data_expert.md`（~80行）**：
```
← 七、数据/信息提取清单（完整）
← 四、内容逻辑分析（仅数据指标体系 + 数据验证规则）
```

**`analysis_for_writer.md`（~260行）**：
```
← 一、文档基本信息（完整）
← 二、报告结构框架（完整）
← 三、格式规范详解（完整）
← 四、内容逻辑分析（完整）
← 五、可变元素分析（完整）
← 六、表达方式和语言风格（完整）
```

两个文件的并集覆盖一至七章全部有用内容，第八章（验证检查清单，与各指导文档自身的验证清单重复）和第九章（分析说明，纯元信息）不输出。

### 11.4 为什么选择源头拆分而非脚本裁剪

曾考虑用 Python 脚本在 TA 输出后裁剪出消费者文件，但对比两次实际运行结果发现 LLM 输出格式不稳定：

| 格式差异 | 第一次运行 | 第二次运行 |
|---------|-----------|-----------|
| 章节编号 | `## 1.`、`## 2.` | `## 一、`、`## 二、` |
| 章节顺序 | 可变元素在语言风格之前 | 可变元素在语言风格之后 |
| 标题措辞 | "数据提取清单" | "数据/信息提取清单" |

脚本裁剪需要对这些格式变体做正则适配，脆弱且维护成本高。

**源头拆分的优势**：
- 直接在 TA 的输出指导中要求分文件输出，LLM 自然理解哪些内容属于哪个消费者
- 无需对输出格式做正则解析
- 即使章节编号和标题措辞变化，内容分配仍然正确

### 11.5 实施方式

修改 `template_analyst.md`（或其按需加载的步骤文件 `ta8_save_verify.md`）的保存步骤：

1. 在 ta8 步骤中将输出要求改为生成两个消费者文件（不再生成完整版）
2. 在 TodoWrite 的 ta8 项中增加说明
3. 修改 SKILL.md 中 Team Lead 传递给 Data Expert 和 Writer 的文件路径

**对 SKILL.md 的改动**：
- 调用 Data Expert 时传递 `analysis_for_data_expert.md` 而非 `analysis_template.md`
- 调用 Writer 时传递 `analysis_for_writer.md` 而非 `analysis_template.md`

### 11.6 预期收益

| 消费者 | 改造前输入 | 改造后输入 | 上下文减少 |
|--------|-----------|-----------|-----------|
| Data Expert | ~350行 | ~80行 | **-77%** |
| Writer | ~350行 | ~260行 | **-26%** |

注意：此处的减少是**真实的上下文总量减少**，与 Section 5.6 的"注意力位置优化"不同。消费者不会读到完整文件，因此减少的内容不会出现在上下文中。

### 11.7 风险与应对

| 风险 | 应对 |
|------|------|
| TA 忘记生成消费者文件 | ta8 步骤文件中明确列出 2 个输出文件，验证清单增加检查项 |
| 消费者文件内容遗漏 | 两个文件并集应覆盖一至七章，可交叉对照排查 |
| TA 上下文增加（输出2个文件的指导） | 增量约 20-30行指导内容，相对 TA 总上下文可接受 |
