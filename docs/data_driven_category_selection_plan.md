# 数据驱动的分析对象选择 — 改动计划

## 一、背景

### 1.1 问题描述

在端到端测试中发现：生成的 10 月报告的上升/下降警情分析对象与模板（11月）完全一致，没有根据实际数据选择。

**模板（11月）分析的对象**：
- 上升：殴打他人、涉未成年人、金牌港园区
- 下降：交通事故

**10月实际数据**：
- 六大类全部上升（交通 +35.2%、群众紧急求助 +23.9%、治安 +10.6%、纠纷 +9.3%、刑事 +5.6%、其他 +2.2%）
- 下降类别：**无**

**但生成报告仍然写了**：上升→殴打他人/涉未成年人/金牌港园区，下降→交通事故 —— 与实际数据矛盾。

### 1.2 根因分析

```
TA（只看模板，不看数据）
  → analysis_for_data_expert.md 中把模板的具体类别写成了固定清单
  → DE 按固定清单提取数据（不做选择判断）
  → Writer 基于拿到的数据生成报告
  → 报告 = 模板翻版，而非数据驱动
```

核心缺陷：**当前流程中没有任何角色负责"根据实际数据决定分析哪些类别"这个决策步骤**。

- TA 分析模板，不看数据 → 只能列模板中的例子
- DE 按清单提取，不做决策 → 清单给什么就提什么
- Writer 基于数据生成，不回溯 → 数据里有什么就写什么

### 1.3 修复目标

将"选什么类别来分析"的决策权从 TA（模板驱动）转移到 DE（数据驱动）。TA 只负责告诉 DE"怎么选"和"选完后按什么维度分析"。

```
修复前：TA 写死清单 → DE 照单提取 → Writer 照数据写 → 报告=模板翻版
修复后：TA 写选择规则 → DE 先算趋势再选对象再提取 → Writer 按实际趋势组织 → 报告=数据驱动
```

## 二、核心设计

### 2.1 信息流改造

**改造前**：
```
TA → analysis_for_data_expert.md:
       第二层提取清单直接列出具体分析对象（从模板中抄来的固定类别名）
       每个对象的分析维度也是固定列出的

DE → 按固定清单提取 → extracted_data.json 中的第二层类别 = 模板原文的类别

Writer → 基于数据写报告 → 报告内容照搬模板分析对象，与实际数据趋势无关
```

**改造后**：
```
TA → analysis_for_data_expert.md:
       第二层提取清单输出"分析对象选择规则"（如何根据第一层结果选择深入对象）
       + "分析维度模板"（选中对象应按哪些维度提取）
       具体规则由 TA 根据模板逻辑动态生成，不固定

DE → de4 第一层提取
   → de4 末尾：应用 TA 的选择规则，根据实际数据决定分析对象
   → de5 按选中的对象逐个深入提取
   → extracted_data.json 包含选择决策记录

Writer → 读取 extracted_data.json 中的选择记录 → 按实际数据趋势组织报告
```

### 2.2 TA 输出改造：从"固定清单"到"选择规则 + 维度模板"

**核心原则：指导文档只规定机制，具体规则由 TA 动态生成。**

TA 的指导文档（`template_analyst.md`）中只写通用约束：
- 第二层提取清单必须输出为"选择规则 + 维度模板"，而非固定类别清单
- 禁止写死具体类别名

TA 在分析具体模板后，在 `analysis_for_data_expert.md` 中动态生成：
- 选择规则：根据该模板的分类逻辑，描述 DE 如何选择深入分析对象（不同模板的规则不同）
- 维度模板：根据该模板的分析维度特征，描述选中对象应提取哪些维度

**示例对比**（说明不同模板会产出不同规则）：

| 模板类型 | TA 动态产出的选择规则 | TA 动态产出的维度模板 |
|----------|----------------------|----------------------|
| 警情分析报告 | "按环比升降分池，取 top 2-3" | 空间、诱因、手段、时间、管理 |
| 财务季报 | "按增长率排序，取波动最大的 3 个科目" | 部门、时间趋势、预算对比 |
| 教育质量报告 | "按达标率排序，取未达标的学科" | 年级、校区、师资 |

指导文档不预设任何一种规则，只要求 TA 产出规则。

### 2.3 DE 执行改造：增加"应用选择规则"步骤

在现有的 de4（第一层提取）末尾，增加"应用选择规则确定分析对象"的子步骤。de5 根据 de4 的选择结果进行深入提取。

**不新增步骤编号**，合并到 de4 中。

改造前：
```
de4: 第一层提取（总量框架数据）
de5: 第二层提取（按 TA 清单中的固定类别）
```

改造后：
```
de4: 第一层提取 + 应用选择规则
     4.1 提取框架性数据（与之前一致）
     4.2 应用 TA 文件中的"分析对象选择规则"
         - 根据第一层提取结果，按规则选择深入分析对象
         - 记录选择结果和理由到 extracted_data.json
         - 选择结果作为 de5 的输入

de5: 第二层提取（按 de4 选中的对象 + TA 的维度模板）
```

### 2.4 extracted_data.json 输出改造

要求 DE 在第一层数据中增加一个 `分析对象选择` 字段，记录决策过程。

**字段结构不在指导文档中固定**——由 TA 的选择规则决定分组方式。指导文档只要求：
- 必须存在选择记录（对象名 + 选择理由 + 分析维度列表）
- 第二层数据的 key 必须与选择记录中的对象一致

### 2.5 Writer 步骤3 改造

Writer 在规划报告结构时，读取 `extracted_data.json` 中的选择记录来决定动态章节内容：
- 哪些对象需要深入分析，以及归入哪个章节（由数据中的选择记录决定）
- 如果某个分组为空（如无下降类别），对应章节不输出
- 每个分析对象的章节结构遵循 `analysis_for_writer.md` 中的结构框架

#### 问题描述

测试中 Writer 生成的代码出现以下写法：
```python
# 错误：Markdown ** 标记作为文本写入 Word 文档
('**', True),
('殴打他人警情', True),
('**', True),

# 错误：run 文本中包含 Markdown ** 标记
('**殴打他人警情**', True),
```

`**` 是 Markdown 语法，python-docx 不会解析它。这些字符会作为字面文本出现在 Word 文档中，而非产生加粗效果。加粗完全由 run 的 `bold` 参数控制。

## 五、验收标准

### 5.1 功能验收

重新执行全流程后：

1. `analysis_for_data_expert.md` 的第二层清单是选择规则，不包含模板中的具体类别名硬编码
2. `extracted_data.json` 中存在分析对象选择记录（对象名 + 选择理由 + 维度列表）
3. 第二层数据中的分析对象是根据第一层实际数据 + 选择规则选出的，与模板原文不同
4. 如果数据中某个分组为空（按规则无符合条件的类别），对应的第二层 key 为空
5. 生成报告中的动态章节与数据中的选择记录一致（选择记录中没有的分组不出现在报告中）

### 5.2 泛化性验收

改动后的流程应对不同数据场景有正确响应。核心原则：**分析对象由"选择规则 + 实际数据"共同决定，不由模板硬编码决定**。

验证方法：同一模板搭配不同数据时，生成的报告应有不同的分析对象和动态章节组合。

### 5.3 回归验收

- TA 输出的 `analysis_for_writer.md` 内容和格式不受影响
- Writer 的格式处理（字体、字号、行距）不受影响
- DE 的数据验证（总和关系、环比计算）不受影响
- DE 的 de1-de3、de6-de8 流程不受影响
