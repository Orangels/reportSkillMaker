## 二、仍存在的问题

### 2.1 问题1（严重）：数据提取维度仍不完整

**现象**：extracted_data.json 为 20KB，仅为 Opus 产出（86KB）的 23%。

**对比清单覆盖率**：

| 清单要求的维度 | 是否提取 | 数据源字段 |
|---------------|---------|-----------|
| 基础数据（总量/分类/环比） | ✅ | 接警报警类别 + 报警时间 |
| 辖区分布（按警情类型） | ✅ | 管辖单位名 |
| 时段分布（按警情类型） | ✅ | 报警时间 |
| 细类分布 | ✅ | 反馈报警细类 |
| **发案区域（地点类型）** | ❌ 缺失 | `最终反馈要素/地点类型` 字段存在但未提取 |
| **发生原因** | ❌ 缺失 | 可从 `反馈报警细类/子类` 提取 |
| **行为手段** | ❌ 缺失 | 需从 `报警内容/反馈信息` 文本分析 |
| **涉刀警情专项** | ❌ 缺失 | 需从 `报警内容` 关键词提取 |
| **涉未成年人专项** | ❌ 缺失 | 需从 `报警内容` 关键词提取 |
| **重点园区专项** | ❌ 缺失 | 需从 `警情地址` 关键词提取 |
| **骚扰警情数量** | ❌ 缺失 | 接警报警类别中可统计 |
| **周末/节假日分析** | ❌ 缺失 | 报警时间可计算 |
| **去年同期数据** | ❌ 全为0 | 数据源有2024年数据（62条） |

**根因分析**：
1. Data Expert 的 de2 渐进式探查停在第2轮（分类字段），未进行第3轮（深入细节）
2. Data Expert 的 de4 未严格对照清单逐项提取，遗漏了模板分析文件中明确要求的特殊分析维度
3. 数据提取质量要求中的"特殊分析项显式提取"规则被跳过

### 2.2 问题2（严重）：格式规范提取严重不完整

**现象**：生成的报告字体、字号、颜色、内嵌加粗全部缺失或不正确。

**Template Analyst 格式规范表缺陷**：

| 格式属性 | 分析文件中的状态 | 应有内容 |
|---------|----------------|---------|
| 字体 | 部分有（宋体/Calibri） | ✅ 基本可用 |
| 字号 | ❌ 大部分标"-" | 应从 XML `w:sz` 提取每种内容类型的实际值 |
| 颜色 | ❌ 完全缺失（无此列） | 应从 XML `w:color` 提取，特别是红头红字 |
| 加粗 | ✅ 有"是/否" | 基本可用 |
| 内嵌加粗模式 | ❌ 未分析 | 应识别正文中"**关键词**"的 run 拆分模式 |

**影响链**：

```
Template Analyst 字号标"-"、颜色完全缺失
  → Writer 被迫猜字号（22pt/16pt/12pt/11pt 全是猜的）
    → Writer 完全不知道要设颜色（红头红字未被提取）
      → Writer 正文中关键数据的内嵌加粗也没做
        → 最终报告：字号不准确、颜色缺失、关键词加粗缺失
```

**generate_report.py 中的具体缺陷**：
- `set_run_font()` 函数无 `run.font.color.rgb` 设置 → 红头标题不是红色
- 发文单位 22pt、主标题 16pt、一级标题 12pt 均为 Writer 猜测值
- 整段文字作为单个 run 输出，无法实现段内关键词加粗（如"**有效警情**2363起"）

### 2.3 问题3（严重）：报告结构缺少特殊分析章节

**模板结构 vs Kimi 报告结构**：

```
模板：                                  Kimi 报告：
二、上升警情                            二、上升警情
├─ (一) 治安殴打他人（5个分析维度）       ├─ (一) 交通警情（3个分析维度）
│   ├─ 发案区域                         │   ├─ 警情细类
│   ├─ 发生原因                         │   ├─ 发案时段
│   ├─ 行为手段                         │   └─ 辖区分布
│   ├─ 涉刀警情                         └─ （仅1个类别）
│   └─ 辖区分布（含四级标题拆分）
├─ (二) 涉未成年人（4个维度）            三、下降警情
└─ (三) 金牌港重点园区                   ├─ (一) 纠纷警情（2个维度）
                                        ├─ (二) 治安警情（2个维度）
三、下降警情                            └─ (三) 刑事警情（1个维度）
└─ (一) 交通事故（2个维度+周末分析）
```

**差距**：
- 模板每类警情 4-5 个分析维度，Kimi 只有 2-3 个
- 模板有涉未成年人、金牌港园区等特殊专题章节，Kimi 完全没有
- 模板治安殴打有四级标题拆分（(1)殴打警情 (2)涉刀警情），Kimi 没有这个层级
- 根因：上游数据不支持（extracted_data.json 缺少这些维度的数据）

### 2.4 问题4（轻微）：数据准确性

- `去年同期数量` 全为 0，`同比变动` 全为 0.0 — 数据源有 2024 年数据（62条），应尝试计算
- 报告开头"接报各类警情2466起"但"有效警情"2455起，差额11条未说明（应为骚扰警情）

## 三、改进方向

### 3.1 针对 Template Analyst（格式提取强化）

**优先级：P0**

**问题**：格式规范表中字号大部分标"-"，颜色完全缺失，内嵌加粗模式未分析。

**改进措施**：在 `template_analyst.md` 中强化格式提取要求：

1. **格式规范表禁止出现"-"**
   - 必须从 XML 中提取每种内容类型的实际 `w:sz` 值（half-points → pt 换算）
   - 如果 XML 中确实没有显式设置，标注"继承默认值 Xpt"而非"-"

2. **增加"颜色"列**
   - 格式规范表必须包含颜色列
   - 从 XML `w:color` 属性提取实际颜色值（如红头 `w:color="FF0000"`）
   - 无颜色设置时标注"默认黑色"

3. **分析正文内嵌加粗模式**
   - 识别模板正文段落中哪些关键词使用了加粗 run
   - 提取加粗关键词的规律（如"警情类别名称"加粗、"数字+单位"加粗等）
   - 在分析文件中增加"内嵌加粗规则"章节

4. **交叉验证检查项**
   - 在检查清单中增加：`格式规范表中无"-"值`
   - 在检查清单中增加：`颜色列已填写`
   - 在检查清单中增加：`内嵌加粗模式已分析`

### 3.2 针对 Data Expert（数据提取深度强化）

**优先级：P0**

**问题**：探查深度不够（2轮 vs 要求3轮），未逐项对照清单，特殊分析项被静默跳过。

**改进措施**：

1. **强化 de2 第3轮探查要求**
   - 第3轮必须输出：`最终反馈要素/地点类型` 的值分布、`反馈报警子类` 的值分布
   - 第3轮必须确认：是否有字段支持"涉刀"、"涉未成年人"等关键词筛选
   - 考虑在 de2 TodoWrite 项中列出第3轮必须探查的具体字段名

2. **强化 de4 清单对照机制**
   - 要求 Data Expert 在提取前打印完整清单
   - 要求逐项标注"已提取 / 未提取（原因：数据源不支持）/ 未提取（原因：字段质量差）"
   - 禁止静默跳过任何清单项

3. **特殊分析项强制检查**
   - 提取完成后，必须显式检查：涉刀警情、涉未成年人、重点区域等特殊分析项
   - 数据源不支持时，在 extracted_data.json 中写入 `"涉刀警情": {"status": "数据源不支持", "reason": "..."}` 而非直接省略

4. **提高文件大小阈值或改用语义验证**
   - 当前 10KB 阈值过低（20KB 通过但远不够）
   - 方案A：提高到 30-40KB
   - 方案B（更好）：检查 JSON 是否包含清单要求的所有顶层 key

### 3.3 针对 Writer（格式应用强化）

**优先级：P1**（依赖 3.1 的 Template Analyst 改进）

**问题**：颜色缺失、字号靠猜、正文内嵌加粗缺失。

**改进措施**：

1. **颜色设置**
   - 在 Writer 指导文档中增加颜色设置要求
   - 在 python-docx 编码规范中增加颜色设置示例代码：
     ```python
     from docx.shared import RGBColor
     run.font.color.rgb = RGBColor(0xFF, 0x00, 0x00)  # 红色
     ```

2. **内嵌加粗实现**
   - 在 Writer 指导文档中增加"段内多 run 输出"要求
   - 正文段落中的关键词需要拆分为独立的 run 并设置 bold
   - 示例：`"我局共接报"` + `"有效警情"(bold)` + `"2363起"` = 3个 run

3. **字号来源验证**
   - Writer 必须从分析文件的格式规范表中读取字号值
   - 如果格式规范表中某项字号缺失，Writer 应报告问题而非自行猜测

### 3.4 针对主流程验证（SKILL.md）

**优先级：P2**

**改进措施**：

1. **步骤4验证增强**
   - 除文件大小检查外，增加"关键维度存在性检查"
   - 读取 extracted_data.json，检查是否包含模板分析清单要求的顶层 key
   - 如果缺少关键维度，要求 Data Expert 补充

2. **步骤3验证增强**
   - 检查 analysis_template.md 中格式规范表是否包含字号和颜色信息
   - 如果格式信息不完整，要求 Template Analyst 补充

## 五、总结

V1 优化（TodoWrite 动态加载机制）解决了**流程层面的核心问题**：Data Expert 不再失败、角色不再错位、基本的验证机制生效。但在**质量层面**仍有两个主要瓶颈：

1. **格式维度**：Template Analyst 的格式提取不完整 → Writer 无法正确设置格式
2. **数据维度**：Data Expert 的清单对照和探查深度不够 → 报告内容深度不足

下一轮优化（V2）应聚焦这两个质量瓶颈，通过强化指导文档中的格式提取要求和数据清单对照机制来提升产出质量。
