# 文本数据提取能力方案

## 问题

当前 DE/DE-deep 只提取可统计数据（总量、环比、分布等），数据源中的文本字段（如反馈内容、处理意见、案件描述等）被完全忽略。导致 Writer 生成的报告：
- 小结和分析只有数字维度，缺少具体内容的定性分析
- 建议章节空洞，无法引用具体案例或归纳具体问题
- 报告深度低于原模板

## 根因

1. DE-deep 多处"禁止纯文字描述"规则误伤了合法的文本数据提取
2. TA 提取清单只覆盖统计指标，无文本数据维度
3. DE 探查和提取步骤只关注可统计字段
4. 全链条缺少文本数据的验证机制

## 修改方案

### 1. TA（template_analyst.md）

**步骤7 提取清单**：增加第四部分——文本数据需求

在清单输出格式中，在现有三层之后增加：

```
#### 第四部分：文本数据需求
TA 必须从模板中识别需要定性内容支撑的章节（如定性分析、典型案例、原因归纳、工作建议等），
并列出需要从数据源提取的文本数据类型。

输出格式：
  - [章节/功能]：需要 [文本数据类型]（如关键词统计、典型记录抽样、文本分类归纳）
  - 如果模板无定性分析需求，标注"无"并说明判断依据

文本数据提取方式参考（DE 根据数据源能力选择）：
  - 关键词/高频词统计：对文本字段提取高频词及频次
  - 典型记录抽样：按条件筛选具有代表性的记录
  - 文本分类归纳：对文本内容按主题或类型归类统计
```

**步骤7 TodoWrite（ta7）**：补充"文本数据需求"提醒

**检查清单**：增加
- `- [ ] 提取清单包含文本数据需求部分（有具体需求或标注"无"+判断依据）`

### 2. DE（data_expert.md）

**关键原则2**：扩展数据概念

在现有"宁多勿少"之后补充：
```
- **数据 = 统计数据 + 文本数据**：不要只提取数字，文本字段（如内容描述、处理意见、备注等）同样是有价值的数据源
```

**步骤2（de2）探查**：第三轮深入细节时增加文本字段识别

在步骤2或渐进式探查要求中补充：
```
第三轮增加：识别数据源中的文本字段，评估其分析价值（如是否可做关键词统计、是否适合抽样）
```

**步骤5（de5）提取**：如果 TA 清单有文本数据需求，按需求提取

在步骤5中补充：
```
如果 TA 清单第四部分有文本数据需求，对相关文本字段执行对应的提取方式（关键词统计、典型记录抽样、文本分类归纳等），结果作为分析对象的子字段输出
```

**步骤6（de6）验证**：清单对照表覆盖文本数据维度

无需单独修改——de6 已要求"逐项列出清单中第一二层的每个维度"，只要 TA 清单包含文本数据部分，de6 自然会覆盖。

**TodoWrite 更新**：
- de2：补充"识别文本字段及分析价值"
- de5：补充"含文本数据需求的维度"

### 3. DE-deep（data_expert_deep.md）

**核心约束1 修正**："禁止纯文字描述"规则精确化

将：
```
- ❌ 禁止用文字描述代替结构化数据
```
改为：
```
- ❌ 禁止用纯文字总结代替应提取的统计数据（如写"交通类占比32%"而不提取实际数值）
- ✅ 允许文本字段的结构化提取结果（如关键词列表、典型记录、文本分类统计）
```

**核心约束3 产出标准**修正：

将"主动发现每条"的标准从：
```
- 必须是结构化数据（JSON 节点含实际统计数值），禁止纯文字描述
```
改为：
```
- 必须是结构化数据（JSON 节点含实际统计数值或结构化文本提取结果），禁止未经加工的纯文字描述
```

**步骤4（dd4）主动发现**：覆盖文本字段

在步骤4的三类检查中补充：
```
- **文本字段利用**：数据源中有分析价值的文本字段是否已做结构化提取（关键词统计、分类归纳等）
```

**TodoWrite 更新**：
- dd4：补充"文本字段结构化提取"

### 4. SKILL.md

**步骤3验证**：增加文本数据需求检查

在 TA 产出语义验证中增加：
```
- [ ] 提取清单包含文本数据需求部分（有具体需求或标注"无"+判断依据）
```

**步骤4验证**：无需修改

de6 的清单对照表已覆盖全部维度，如果 TA 清单包含文本数据需求，de6 验证自然会核对。Team Lead 步骤4验证中已有"数据结构为层级化组织"的检查，文本数据提取结果也应符合此要求。
