# DE 拆分方案：解决 de6/de7 步骤跳过问题

## 问题描述

Data Expert 在执行 de1-de9 的过程中，de6（第三层特殊分析项）和 de7（主动发现）经常被跳过或产出空壳数据。

### 典型表现
- 第三层数据为空壳：`{"涉未成年人": {"总数": 0, "分布": {}, "辖区分布": {}}}`
- 主动发现只有文字描述，没有实际统计数据
- "不支持"标注没有具体原因，未尝试过筛选
- 补充数据与已有数据冲突（如辖区分布出现两份不一致的版本）

### 根因分析

**模型能力因素（~60%）：**

1. **注意力衰减** — de1-de5 消耗大量 context（代码+数据），到 de6 时注意力预算已近枯竭
2. **赶进度倾向** — 已完成 5/9 步，模型倾向快速结束剩余步骤
3. **最小阻力路径** — "写空壳 JSON"和"写代码筛选"都能标记完成，模型选前者

**文档因素（~40%）：**

1. **de6/de7 缺乏"强制代码执行"约束** — de4/de5 天然需要写代码（上千条记录的汇总），de6/de7 可以不写代码就过关
2. **de8 是"自己验自己"** — 等于考生自己批改试卷
3. **Team Lead 验证太浅** — 只检查"第三层存在"，空壳 `{}` 也能通过

## 修改方案

### 方案概述

两手并行：**拆分 agent 调用**（解决注意力衰减）+ **文档强化约束**（堵住偷懒路径）+ **Team Lead 验证加深**（外部兜底）。

### 一、Team Lead 调用流程变更（SKILL.md）

**原流程：**
```
Team Lead
├─ 调用 TA subagent         → analysis_template.md
├─ 验证 TA 产出
├─ 调用 DE subagent (de1-de9) → extracted_data.json
├─ 验证 DE 产出
├─ 调用 Writer subagent      → 报告.docx
└─ 最终质量验证
```

**新流程：**
```
Team Lead
├─ 调用 TA subagent              → analysis_template.md
├─ 验证 TA 产出
├─ 调用 DE subagent (de1-de5,de8-de9) → extracted_data.json（第一二层）
├─ 验证第一二层数据
├─ 调用 DE-deep subagent (de6-de7,de8) → 补充第三层+主动发现到 JSON
├─ 验证第三层+主动发现数据
├─ 调用 Writer subagent           → 报告.docx
└─ 最终质量验证
```

**关键设计：**
- DE-deep 由 Team Lead 直接调用（不是 DE 调用子 agent）
- DE-deep 接收三个输入：`analysis_template.md` 路径、`extracted_data.json` 路径、数据源路径
- DE-deep 的 context 是干净的，注意力充足
- Team Lead 在 DE 和 DE-deep 之间有一次验证机会

### 二、DE-deep subagent 定义（新增文件）

**文件**：`.claude/agents/data-expert-deep.md`

**职责**：在 DE 已完成第一二层数据的基础上，专注执行第三层特殊分析项提取和主动发现。

**输入**：
- `analysis_template.md` — 找到第三层清单和特殊分析项定义
- `extracted_data.json` — 已有的第一二层数据（读取但不修改已有内容）
- 数据源文件路径

**输出**：
- 更新后的 `extracted_data.json`（在已有基础上追加第三层和主动发现）

### 三、DE 文档强化约束（data_expert.md 修改）

#### 3.1 步骤6（第三层）强化

**现状问题**：只说"逐项检查、独立统计、标注不支持"，没有强制写代码。

**修改要点**：
- 必须为每个特殊项编写并执行 Python 脚本进行筛选
- 产出标准量化：每个特殊项要么有实质数据（≥1个非空非零子字段），要么附带尝试过的筛选代码和条件
- 禁止空壳产出：`{"总数": 0, "分布": {}}` 不算"已提取"
- 禁止不写代码就标注"数据源不支持"——必须先执行代码尝试筛选，确认无结果后才能标注
- "不支持"标注必须包含：尝试了哪些列、用了什么筛选条件、为什么没有匹配记录

#### 3.2 步骤7（主动发现）强化

**现状问题**：只说"逐列扫描、维度对称、交叉分析"，可以写文字总结就过关。

**修改要点**：
- 必须编写 Python 脚本执行逐列扫描，输出每列的唯一值数量和样本值
- 对识别出的有价值维度，必须执行代码提取实际数据
- 产出必须是结构化数据（JSON 节点），禁止纯文字描述
- 禁止出现 `"数据规律": ["xxx占比32%", "yyy下降2.4%"]` 这类纯文字条目（这些是对已有数据的复述，不是"发现"）
- 真正的主动发现示例：星期分布、响应时间统计、专项分析（盗窃/骚扰）、派出所对比排名等

#### 3.3 步骤8（验证）强化

**修改要点**：
- 第三层验证：逐项检查每个特殊分析项是否有实质数据或有效的"不支持"标注（含代码证据）
- 主动发现验证：确认每条发现都是结构化数据，非纯文字复述
- 新增"反面检查"：搜索 JSON 中所有值为 `{}`、`[]`、`0` 的节点，逐一确认是真实数据还是空壳

### 四、Team Lead 验证加深（SKILL.md 修改）

#### 4.1 DE 产出验证（第一二层）

在现有验证基础上增加：
- 检查分析对象选择记录是否存在且合理
- 检查第二层是否覆盖了所有选中的分析对象
- 检查不存在重复/冲突的数据节点

#### 4.2 DE-deep 产出验证（新增）

```
验证清单：
- [ ] 第三层每个特殊项有实质数据，或"不支持"标注包含具体筛选条件
- [ ] 不存在空壳节点（值为 {} 或所有子字段为 0）
- [ ] 主动发现是结构化数据，非纯文字描述
- [ ] 补充数据与已有第一二层数据不冲突（无重复字段、无数值矛盾）
- [ ] JSON 整体可解析，结构完整
```

**失败处理**：
- 空壳或纯文字 → 指明具体问题，重新调用 DE-deep（最多重试1次）
- 数据冲突 → 指明冲突节点，重新调用 DE-deep 修正
